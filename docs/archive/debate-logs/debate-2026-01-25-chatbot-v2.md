## 토론 전체 기록

### ROUND 1 / 20
**FOCUS: MVP 범위 및 핵심 목적 정의**

#### [긍정적 기획자]
* **반응**: MVP 범위가 명확하게 정의되어 있습니다. 장소 추천 + 여행 Q&A + 대화 이력 저장이라는 핵심 3가지에 집중한 것은 현명한 선택입니다. 1.5~2주라는 일정도 현실적입니다.
* **근거**:
  - 프로젝트 컨텍스트(destination, country) 자동 반영으로 기존 시스템과의 통합이 자연스럽습니다
  - Gemini Flash 선택은 비용 효율적이며 향후 전체 마이그레이션 계획과 일관성이 있습니다
  - 일일 50회 제한은 비용 통제와 서비스 안정성 확보에 적절합니다
* **PRD 제안**: 현재 상태 유지. MVP 범위가 적절히 정의됨

#### [부정적 기획자]
* **반응**: MVP 범위는 합리적이나, "제휴 마케팅 확장을 통한 수익 모델 기반"이라는 목표와 현재 MVP 기능 간의 연결고리가 불분명합니다. 또한 "한국어 전용"이라는 제약이 글로벌 확장 가능성을 제한합니다.
* **리스크**:
  - 수익 모델 검증 없이 비용만 발생하는 기능이 될 수 있음
  - 일본 여행 추천에서 한국어만 지원 시 현지 정보 정확도 저하 가능성
  - "프로젝트 상세 페이지만"이라는 제한이 사용자 접근성을 낮출 수 있음
* **보강 제안**:
  - Phase 2에서 수익 모델 검증 마일스톤 추가 필요
  - 일본어/영어 장소명 표기 지원 방안 명시

### ROUND 2 / 20
**FOCUS: 성공 지표 및 측정 가능성**

#### [긍정적 기획자]
* **반응**: 성공 지표가 구체적이고 측정 가능합니다. TTFB P50 < 1초, 장소 추가 전환율 > 30%, Geocoding 성공률 > 95%는 모두 기술적으로 추적 가능한 수치입니다.
* **근거**:
  - APM 로그 기반 TTFB 측정은 Vercel Analytics로 쉽게 구현 가능
  - 장소 추가 전환율은 이벤트 트래킹으로 정확히 측정 가능
  - Geocoding 성공률은 API 응답 기반으로 자동 집계 가능
* **PRD 제안**: 현재 지표 유지, 다만 "일일 활성 사용자" 목표치 추가 권장 (예: 출시 1개월 후 DAU 50명)

#### [부정적 기획자]
* **반응**: 지표는 측정 가능하나, 사용자 만족도나 추천 품질에 대한 정성적 지표가 없습니다. 또한 "장소 추가 전환율 30%"의 근거가 불분명합니다.
* **리스크**:
  - TTFB만 측정하고 전체 응답 완료 시간(Time to Last Byte)은 측정하지 않음
  - AI 추천 품질에 대한 사용자 피드백 수집 메커니즘 부재
  - 30% 전환율 목표의 산정 근거가 없어 목표 달성 여부 판단 곤란
* **보강 제안**:
  - 사용자 피드백 메커니즘 추가 (예: 추천 유용성 평가 버튼)
  - 전환율 목표 산정 근거 명시 또는 "측정 후 베이스라인 설정"으로 변경

### ROUND 3 / 20
**FOCUS: 기존 시스템과의 통합 범위**

#### [긍정적 기획자]
* **반응**: 프로젝트 상세 페이지에만 챗봇을 배치하는 결정은 컨텍스트 집중과 구현 복잡도 최소화 측면에서 탁월합니다. 기존 Place 모델과의 연결도 잘 설계되어 있습니다.
* **근거**:
  - `ChatMessage.createdPlaces` 관계로 AI 추천과 장소 생성 이력 추적 가능
  - 프로젝트 컨텍스트 자동 주입으로 사용자 입력 부담 감소
  - 기존 Geocoding 인프라(4단계 fallback) 재활용 가능
* **PRD 제안**: 기존 `geocodePlaceWithFallback()` 함수 재사용 명시

#### [부정적 기획자]
* **반응**: 기존 시스템 통합 설계는 적절하나, "그룹 공유: 개별"이라는 결정이 협업 기능과 충돌합니다. 같은 프로젝트 멤버가 다른 멤버의 AI 추천을 볼 수 없다는 것은 협업 경험을 저해합니다.
* **리스크**:
  - 그룹 프로젝트에서 멤버 간 AI 추천 공유 불가로 중복 질문 발생
  - 기존 장소 추가 기능과 AI 추가 장소의 UI/UX 일관성 확보 필요
  - `createdByMessageId` 관계에서 메시지 삭제 시 Place 참조 깨짐 가능성
* **보강 제안**:
  - 그룹 공유 옵션 추가 검토 (Phase 2로 이관 가능)
  - 메시지 삭제 시 `SetNull` 정책 적용 확인 (이미 적용됨 - OK)

### ROUND 4 / 20
**FOCUS: 사용자 스토리 US-001 (장소 추천)**

#### [긍정적 기획자]
* **반응**: US-001의 수락 기준이 구체적이고 테스트 가능합니다. "이미 프로젝트에 있는 장소 추천 시 '이미 추가됨' 표시"는 중요한 엣지 케이스를 잘 다루고 있습니다.
* **근거**:
  - 장소 카드에 표시할 정보(장소명, 주소, 설명, Google Maps 링크)가 명확함
  - "3문장 이내" 설명 제한으로 AI 응답 길이 통제
  - Geocoding API 연동 명시로 구현 방향 명확
* **PRD 제안**: 장소 카드에 "카테고리" 필드 추가 권장 (기존 Place 모델과 일관성)

#### [부정적 기획자]
* **반응**: 수락 기준은 좋으나, 여러 중요한 시나리오가 누락되어 있습니다. 특히 "추천 결과가 0개인 경우"와 "사용자가 추천을 거부하는 경우"에 대한 처리가 없습니다.
* **리스크**:
  - AI가 장소를 찾지 못했을 때의 UX 정의 없음
  - 같은 장소를 다른 이름으로 추천받았을 때 중복 감지 방법 불명확
  - "다른 추천" 요청(US-005)과 연계된 구체적 구현 방법 없음
* **보강 제안**:
  - "추천 결과 없음" 시나리오 수락 기준 추가
  - 기존 DuplicateDetectionService 활용 방안 명시

  - 여행 관련 질문에 집중하여 off-topic 응답 방지

* **반응**: 카테고리 정의가 모호합니다. "환율"은 실시간 데이터가 필요하고, "날씨"도 마찬가지인데, Gemini가 실시간 정보를 제공할 수 없습니다.
* **리스크**:
  - 환율 질문에 오래된 정보 제공 시 사용자 신뢰도 하락
  - 날씨 예보 요청 시 정확한 답변 불가
  - 4개 카테고리 외 질문 시 "답변할 수 없습니다" 처리 방법 불명확
* **보강 제안**:
  - 실시간 정보 필요 질문 시 외부 링크 제공 방식 명시
  - off-topic 질문 처리 정책 추가 (예: "여행 관련 질문만 도와드릴 수 있어요")

### ROUND 7 / 20
**FOCUS: 데이터 모델 설계**

#### [긍정적 기획자]
* **반응**: 데이터 모델이 깔끔하고 확장 가능합니다. `@@unique([projectId, userId])`로 프로젝트+사용자당 1개 세션을 강제하는 것은 명확한 비즈니스 규칙 구현입니다.
* **근거**:
  - `ChatMessage.contentType` 필드로 텍스트/장소/에러 응답 구분
  - `metadata` JSON 필드로 유연한 데이터 저장
  - `onDelete: Cascade`로 프로젝트 삭제 시 자동 정리
* **PRD 제안**: 현재 설계 유지, 인덱스 전략 적절함

#### [부정적 기획자]
* **반응**: 데이터 모델에 몇 가지 누락이 있습니다. `ChatMessage.content`가 `@db.Text`로 64KB 권장이지만, 스트리밍 중단 시 부분 응답 저장을 위한 `isComplete` 플래그가 없습니다.
* **리스크**:
  - 스트리밍 중단 시 부분 응답과 완료 응답 구분 불가
  - `ChatUsage.date`가 어떤 타임존 기준인지 불명확 (KST?)
  - 분당 rate limit 추적을 위한 테이블/필드 없음
* **보강 제안**:
  - `ChatMessage`에 `status` 필드 추가 (`streaming` | `complete` | `error`)
  - `ChatUsage` 타임존 명시 또는 UTC 저장 후 KST 변환 정책 명시

### ROUND 8 / 20
**FOCUS: API 설계 - 채팅 API (POST /api/projects/[id]/chat)**

#### [긍정적 기획자]
* **반응**: SSE 기반 스트리밍 API 설계가 적절합니다. `X-Request-Id`와 `messageId`로 중복 전송 방지 메커니즘을 갖추고 있습니다.
* **근거**:
  - SSE는 WebSocket보다 구현 간단하고 Vercel 서버리스 환경에 적합
  - `type` 필드로 텍스트/장소/완료/에러 구분 명확
  - `messageId` 클라이언트 생성으로 idempotency 보장
* **PRD 제안**: SSE 연결 타임아웃 정책 명시 (Vercel 60초 제한 고려)

#### [부정적 기획자]
* **반응**: API 설계에서 몇 가지 엣지 케이스가 누락되었습니다. SSE 연결이 끊어졌을 때 재연결 프로토콜이 없고, 부분 응답 복구 방법도 불명확합니다.
* **리스크**:
  - 네트워크 불안정 시 SSE 재연결 로직 필요
  - 60초 타임아웃 도달 시 응답 중단 처리 방법 없음
  - `Authorization: Bearer <session_token>` 형식이 NextAuth 세션 쿠키와 불일치 (쿠키 기반 인증 사용 시)
* **보강 제안**:
  - SSE 재연결 및 `Last-Event-ID` 지원 검토
  - API 인증 방식 명확화 (쿠키 vs Bearer 토큰)
